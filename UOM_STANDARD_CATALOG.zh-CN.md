# 单位标准目录（当前执行基线）

## 目标

这份文件用于整理 `myapp` / `myapp-mobile` 当前与后续应统一采用的正式单位主数据，解决单位内容混乱、中英文混用、临时测试单位残留的问题。

这份目录的设计原则是：

- 系统内部使用稳定编码作为 `name`
- 前台业务展示统一使用中文 `display_name`
- `symbol` 仅作为缩写 / 紧凑展示辅助，不作为主识别字段
- 常规业务页面、订单、发票、PDF 默认优先显示 `display_name`
- 只有确有需要时，才在紧凑位置显示 `symbol`

## 字段规范

推荐约定：

- `name`
  - 系统内部稳定编码
  - 建议保持英文或行业常见稳定缩写
  - 一旦投入业务引用后，不应随意修改

- `uom_name`
  - 单位主名称
  - 在当前阶段可与 `name` 保持一致

- `display_name`
  - 中文正式展示名
  - 商品页、订单页、发票页、PDF、打印单据默认使用这个字段

- `symbol`
  - 单位缩写 / 符号
  - 仅用于紧凑显示、表格列头、报表简写、标签补充

- `must_be_whole_number`
  - 是否必须整数
  - 用于限制录入数量是否允许小数

## 业务规则建议

- 件数类 / 包装类单位通常设为 `must_be_whole_number = 1`
  - 例如：件、箱、包、瓶、桶、卷、套、板

- 重量 / 体积 / 长度类单位通常设为 `must_be_whole_number = 0`
  - 例如：千克、克、升、毫升、米、厘米

- 正式单据默认显示 `display_name`
  - 例如：`千克`、`升`、`箱`

- `symbol` 仅在这些场景使用：
  - 列宽较小的表格
  - 紧凑标签
  - 报表列头
  - 辅助括注，例如：`千克 (kg)`

## 建议不要继续保留的单位类型

这些不是“永远不能存在”，而是如果没有明确业务用途，不建议作为正式主数据长期保留：

- 临时测试单位
  - 例如：`HTTP-UOM-*`

- 含糊不清的中英文混合写法
  - 例如：`box箱`
  - 例如：`kg千克`

- 语义重复但没有统一规则的近义单位
  - 例如：`个`、`件`、`只`、`pcs` 同时无约束地混用

- 只有英文无中文展示策略的历史单位
  - 例如：页面只能显示 `Nos`、`Case`、`Yard`

### 数据库中已存在但当前不建议列入第一批正式业务目录的单位

这些单位不一定错误，但对你们当前业务不是第一优先级：

- 英制或海外扩展单位
  - `Acre`
  - `Square Yard`
  - `Mile`
  - `Pint (US)`
  - `Gallon (UK)`

- 科学 / 工业 / 物理单位
  - `Ampere`
  - `Pascal`
  - `Tesla`
  - `Watt`
  - `Joule`
  - `Newton`

- 各类复合单位
  - `Kilometer/Hour`
  - `Meter/Second`
  - `Gram/Litre`
  - `Pound/Cubic Foot`

这些单位可以暂时保留在系统里，但不建议直接放进你们“正式业务单位白名单”里。

### 明显需要清理或隔离的现有单位

- 测试单位
  - `HTTP-RANGE-UOM-*`
  - `HTTP-UOM-RAW-*`
  - `HTTP-UOM-SYM-*`

- 历史上只为接口测试或临时验证创建的单位
  - 这类单位不应继续出现在正式业务选择列表中

### 当前数据库盘点后的建议

基于这次实际盘点，比较稳的做法是：

1. 第一批正式目录优先复用数据库里已经存在的常用单位
   - 例如：`Nos`、`Box`、`Kg`、`Litre`、`Meter`、`Day`

2. 先补齐这些现有单位的中文展示名和 symbol 规范

3. 将测试单位从正式使用范围中排除

4. 对 ERPNext 默认带来的科学 / 工业单位先不删除
   - 但后续应避免在移动端常规选择器中直接暴露它们

## 建议清洗顺序

如果后面要真正落地清洗现有单位，建议按这个顺序做：

1. 冻结新增临时单位
2. 先整理正式单位白名单
3. 标记并筛出测试单位、重复单位、历史英文单位
4. 先统一 `display_name`
5. 再逐步梳理商品、订单、采购中的单位引用
6. 对无引用单位做合并或删除
7. 对有引用单位优先停用，不直接暴力改名
